## 1. 프로젝트 개요

### 1.1 배경
평생교육원은 매 학기 교육과정 운영성과를 점검하고 다음 학기 계획에 반영하기 위해 수강생 만족도 조사를 실시한다. 기존 방식(종이 설문 또는 외부 설문 도구)은 다음과 같은 한계가 있다.

- 외부 설문 도구는 평생교육원 고유의 디자인·문항 구조를 자유롭게 반영하기 어렵다.
- 시니어 응답자에게는 글씨가 작거나 조작이 복잡해 응답 중도 이탈이 발생한다.
- 응답 데이터가 외부에 저장되어 자체 분석·재활용이 제한된다.

이를 해결하기 위해 평생교육원이 직접 운영·확장할 수 있는 **자체 설문 시스템**을 구축한다.

### 1.2 목표
1. 시니어 수강생이 **별도 설명 없이도** 모바일에서 끝까지 응답을 완료할 수 있는 시스템을 만든다.
2. 한 번 만든 시스템에 **여러 설문을 계속 추가**할 수 있도록 데이터 구조와 코드를 설계한다.
3. 관리자가 응답 결과를 **대시보드로 확인하고 Excel/CSV로 내보낼** 수 있게 한다.
4. 개인식별정보를 수집하지 않아 **개인정보 부담 없이** 운영한다.

### 1.3 범위

**1차(이번 학기) 범위에 포함**
- 만족도 설문 1종 운영 (응답 화면, 응답 저장)
- 관리자 결과 대시보드 + Excel/CSV 내보내기
- Railway 배포, PostgreSQL 연동, GitHub 소스 관리

**향후 확장 (이번 범위에는 미포함, 구조만 대비)**
- 설문을 화면에서 직접 만드는 설문 빌더
- 중복 응답 방지를 위한 토큰/링크 발급
- 다국어, SMS 자동 발송 연동 등

## 2. 사용자 정의

| 구분 | 사용자 | 주요 행동 | 특징 |
|------|--------|-----------|------|
| 응답자 | 평생교육원 수강생 | 문자로 받은 URL 접속 → 설문 응답 | **시니어 비중 높음**, 모바일 사용, 디지털 친숙도 낮음 |
| 관리자 | 평생교육원 담당자 | 결과 확인, 데이터 내보내기, 설문 기간 관리 | PC 사용, 통계·분석 목적 |

> **설계 우선순위**: 응답자(시니어) 경험을 최우선으로 하고, 관리자 화면은 기능 위주로 단순하게 구성한다.

## 3. 시스템 아키텍처

### 3.1 전체 구성

```
[수강생 휴대폰]                 [관리자 PC]
      │ (공개 URL)                  │ (관리자 로그인)
      ▼                             ▼
┌─────────────────────────────────────────────┐
│              Railway (호스팅)                  │
│  ┌─────────────────────────────────────────┐ │
│  │  웹 애플리케이션 (Next.js)                 │ │
│  │   - 응답자용 설문 페이지                    │ │
│  │   - 관리자용 대시보드                       │ │
│  │   - API (응답 저장 / 집계 / 내보내기)        │ │
│  └─────────────────────────────────────────┘ │
│                    │                          │
│                    ▼                          │
│  ┌─────────────────────────────────────────┐ │
│  │  PostgreSQL (Railway 관리형 DB)           │ │
│  └─────────────────────────────────────────┘ │
└─────────────────────────────────────────────┘
      ▲
      │ 배포 (자동)
┌──────────────┐
│   GitHub      │  소스 코드 / 버전 관리 / Railway 자동 배포 연동
└──────────────┘
```

### 3.2 동작 흐름
1. 관리자가 GitHub에 코드를 푸시하면 Railway가 자동으로 빌드·배포한다.
2. 수강생은 문자로 받은 공개 URL에 접속해 설문에 응답한다.
3. 응답은 즉시 PostgreSQL에 저장된다.
4. 관리자는 관리자 페이지에서 집계 결과를 확인하고 데이터를 내보낸다.

## 4. 기술 스택

| 영역 | 선택 | 선택 이유 |
|------|------|-----------|
| 프레임워크 | **Next.js (App Router) + TypeScript** | 응답자 화면과 관리자/API를 한 프로젝트에서 처리, Railway 배포 용이 |
| 스타일링 | **Tailwind CSS** | 시니어용 큰 글씨·큰 버튼 등 디자인 토큰을 일관되게 관리 |
| DB | **PostgreSQL (Railway 관리형)** | 안정적인 관계형 DB, 설문 확장 구조에 적합 |
| ORM | **Prisma** | 스키마 기반 관리, 마이그레이션·타입 안정성 |
| 내보내기 | **ExcelJS / CSV** | 관리자 Excel·CSV 다운로드 |
| 차트 | **Recharts 또는 Chart.js** | 대시보드 시각화 |
| 호스팅/CI | **Railway + GitHub 연동** | 푸시 시 자동 배포 |

> 위 스택은 권장안이며, 팀 익숙도에 따라 백엔드를 별도 분리(예: Node/Express + 정적 프론트)할 수도 있다. 단, 운영 단순성을 위해 **단일 Next.js 프로젝트**를 권장한다.

## 5. 데이터베이스 설계

여러 설문을 계속 추가할 수 있도록 **설문(survey) → 문항(question) → 응답(response) → 응답상세(answer)** 의 범용 구조로 설계한다. 만족도 설문은 이 구조의 첫 번째 데이터일 뿐이다.

### 5.1 테이블 구조

**survey (설문)**

| 컬럼 | 타입 | 설명 |
|------|------|------|
| id | PK | 설문 고유 ID |
| title | text | 설문명 (예: 2026학년도 1학기 만족도 조사) |
| description | text | 설문 안내문 |
| status | enum | draft / open / closed |
| start_at | timestamp | 응답 시작일 |
| end_at | timestamp | 응답 종료일 |
| created_at | timestamp | 생성일 |

**question (문항)**

| 컬럼 | 타입 | 설명 |
|------|------|------|
| id | PK | 문항 ID |
| survey_id | FK | 소속 설문 |
| section | text | 문항 그룹 (A 응답자 정보 / B 강의·강사 / C 환경·운영 / D 종합 / E 자유의견) |
| order_no | int | 표시 순서 |
| code | text | 문항 코드 (A1, B1 …) |
| text | text | 문항 내용 |
| type | enum | short_text / single_choice / multi_choice / scale_5 / long_text |
| required | bool | 필수 여부 |
| guide | text | 보조 안내문/예시 |

**option (선택지)** — 객관식·척도형 문항의 보기

| 컬럼 | 타입 | 설명 |
|------|------|------|
| id | PK | 선택지 ID |
| question_id | FK | 소속 문항 |
| order_no | int | 표시 순서 |
| label | text | 보기 텍스트 (예: 매우 만족) |
| value | int/text | 저장 값 (척도 1~5 등) |

**response (응답 1건 = 응답자 1명의 제출)**

| 컬럼 | 타입 | 설명 |
|------|------|------|
| id | PK | 응답 ID |
| survey_id | FK | 소속 설문 |
| submitted_at | timestamp | 제출 시각 |
| user_agent | text | 기기/브라우저(통계용, 개인식별 아님) |

**answer (문항별 응답값)**

| 컬럼 | 타입 | 설명 |
|------|------|------|
| id | PK | 응답상세 ID |
| response_id | FK | 소속 응답 |
| question_id | FK | 대상 문항 |
| value_text | text | 단답/장문/선택 텍스트 |
| value_number | int | 척도/선택 값 |

> **개인정보 미수집 원칙**: 성명·연락처·학번 등 개인식별정보 컬럼은 두지 않는다. `user_agent`는 PC/모바일 비율 등 통계 목적으로만 쓴다.

### 5.2 확장성 포인트
- 새 설문은 `survey` 한 행 + `question`/`option` 추가만으로 운영된다. 코드 수정 없이 데이터로 설문이 늘어난다.
- 문항 `type`만 늘리면(예: 순위형, 날짜형) 새로운 유형도 수용할 수 있다.

## 6. 설문 문항 구성 (1차: 만족도 조사)

업로드된 설문지(안) 기준. 총 23문항, 약 3~5분 소요.

### 6.1 기본 설정

| 항목 | 내용 |
|------|------|
| 설문명 | 2026학년도 1학기 평생교육원 수강생 만족도 조사 |
| 응답대상 | 1학기 수강생 전체 (개설 강좌 102개, 등록인원 1,218명 기준) |
| 응답기간 | 2026. 6. 22.(월) ~ 2026. 6. 30.(화) |
| 응답방식 | 문자 안내 → 온라인 URL 접속 → 모바일/PC 응답 |
| 척도 | 5점 척도 (① 매우 불만족 ~ ⑤ 매우 만족) |
| 개인정보 | 개인식별정보 미수집 |

### 6.2 문항 구성표

**A. 응답자 기본 정보**

| 코드 | 문항 | 유형 | 보기/안내 | 필수 |
|------|------|------|-----------|------|
| A1 | 수강하신 강좌명을 입력해 주세요 | 단답형 | 예: 수필창작, 태극권, AI 강좌 등 | 필수 |
| A2 | 수강 과정 구분 | 객관식 | 생활교양 / 전문자격 / 기타 | 필수 |
| A3 | 수강 시간대 | 객관식 | 주간 / 야간 / 기타 | 필수 |
| A4 | 수강하게 된 주된 목적 | 복수선택 | 취미·여가 / 건강관리 / 자기계발 / 자격취득 / 재취업·창업 / 지인추천 / 기타 | 선택 |

**B. 강의·강사 (5점 척도)**

| 코드 | 문항 | 필수 |
|------|------|------|
| B1 | 강의 내용은 수강 목적과 기대에 부합했다 | 필수 |
| B2 | 강의 난이도와 진도는 적절했다 | 필수 |
| B3 | 강사는 해당 분야의 전문성과 준비성을 갖추었다 | 필수 |
| B4 | 강사의 설명·소통·피드백은 수업 이해에 도움이 되었다 | 필수 |
| B5 | 교재·자료·실습/활동 구성은 학습에 도움이 되었다 | 필수 |

**C. 교육환경·운영 (5점 척도)**

| 코드 | 문항 | 필수 |
|------|------|------|
| C1 | 강의실·기자재 등 교육환경은 수업에 적절했다 | 필수 |
| C2 | 수강신청·등록·안내 등 행정서비스에 만족했다 | 필수 |
| C3 | 교육기간·수업시간·운영방식은 적절했다 | 필수 |

**D. 종합 만족·재수강 (5점 척도)**

| 코드 | 문항 | 척도 | 필수 |
|------|------|------|------|
| D1 | 수강료 대비 교육 내용·운영에 만족했다 | 매우 불만족~매우 만족 | 필수 |
| D2 | 전반적으로 이번 학기 강좌에 만족한다 | 매우 불만족~매우 만족 | 필수 |
| D3 | 향후 평생교육원 강좌를 재수강할 의향이 있다 | 전혀 없음~매우 있음 | 필수 |
| D4 | 평생교육원 강좌를 지인에게 추천할 의향이 있다 | 전혀 없음~매우 있음 | 필수 |

**E. 자유의견 (장문형, 모두 선택)**

| 코드 | 문항 |
|------|------|
| E1 | 가장 만족스러웠던 점 |
| E2 | 개선이 필요한 점 |
| E3 | 2학기 희망 강좌·운영방식 |
| E4 | 추가로 전하고 싶은 의견 |

### 6.3 시작·완료 화면 문구

**시작 화면**
> 안녕하십니까? 충남대학교 평생교육원입니다.
> 2026학년도 1학기 수강생 여러분의 의견을 듣고, 교육과정 운영 개선 및 2학기 교육계획 수립에 반영하고자 만족도 조사를 실시합니다. 응답은 통계 목적으로만 활용되며 개인식별정보는 수집하지 않습니다. 잠시만 시간을 내어 참여해 주시면 감사하겠습니다.

**완료 화면**
> 소중한 의견을 남겨주셔서 감사합니다. 응답 내용은 2026학년도 1학기 성과 점검 및 2학기 교육과정 개선에 적극 활용하겠습니다.

## 7. 기능 명세

### 7.1 응답자 기능

| 기능 | 설명 |
|------|------|
| 설문 시작 화면 | 안내문 + 큰 "설문 시작하기" 버튼 |
| 단계별 응답 | 섹션(A~E)별로 화면을 나눠 한 번에 보이는 문항 수를 줄임 |
| 진행 표시 | 상단에 "5문항 중 2번째" 식의 큰 진행 표시 |
| 5점 척도 입력 | 큰 버튼 + 색상/표정 보조로 직관적 선택 |
| 필수 검증 | 필수 문항 미응답 시 다음으로 넘어가지 못하게 하고 친절히 안내 |
| 이전/다음 이동 | 큰 버튼, 응답 중간 저장(뒤로 가도 입력 유지) |
| 제출 및 완료 | 제출 후 감사 화면, 중복 제출 방지(완료 후 재접속 시 안내) |
| 기간 외 안내 | 응답기간 전/후 접속 시 "현재 응답 기간이 아닙니다" 안내 |

### 7.2 관리자 기능

| 기능 | 설명 |
|------|------|
| 로그인 | 관리자 전용 접근 (아이디/비밀번호, 환경변수 기반 단순 인증) |
| 응답 현황 | 총 응답 수, 기간 내 일자별 응답 추이 |
| 정량 집계 | 전체 평균, 과정별·시간대별 평균, 문항별 만족도 분포, 재수강·추천 의향 |
| 강좌별 보기 | A1(강좌명) 기준 그룹 집계 |
| 정성 응답 | E1~E4 서술형 응답 목록 조회·검색 |
| 데이터 내보내기 | **Excel/CSV** 다운로드 (원자료 + 집계표) |

### 7.3 결과 정리 항목 (설문지 기준)
- **정량분석**: 전체 평균, 과정별 평균, 강좌별 평균, 문항별 만족도, 재수강·추천 의향
- **정성분석**: 좋았던 점 / 개선 요구 / 희망 강좌 / 행정서비스 의견 유형별 분류
- **활용방향**: 강좌 편성, 강사 피드백, 교육환경 개선, 수강신청·안내 체계 개선, 2학기 홍보 반영

## 9. 시니어 중심 UI/UX 디자인 가이드

> 핵심 철학: **"읽기 쉽고, 누르기 쉽고, 헷갈리지 않게."** 한 화면에는 한 가지만 묻고, 모든 요소를 크게.

(상세 기준은 아래 별도 디자인 가이드 문서에도 정리)

### 9.1 핵심 원칙 (요약)
1. **큰 글씨**: 본문 18px 이상, 문항 텍스트 22~24px 이상.
2. **큰 터치 영역**: 모든 버튼·선택지 최소 높이 56px 이상(권장 60px+), 간격 넉넉히.
3. **높은 명도대비**: 본문 텍스트 대비비 WCAG AA(4.5:1) 이상, 핵심 버튼은 더 강하게.
4. **한 화면 한 주제**: 스크롤·선택을 최소화, 섹션별 분할.
5. **명확한 피드백**: 선택 시 색·테두리로 분명히 표시, 진행 상태 항상 표시.
6. **직관적 척도**: 5점 척도에 색상 그라데이션 + 보조 라벨로 의미를 즉시 전달.
7. **간결한 문구**: 어려운 용어·약어 지양, 짧고 친절한 문장.

### 9.2 색상·타이포 (초안)
- 기본 배경: 흰색 / 밝은 회색. 강조색: 충남대 상징색 계열 1색으로 절제.
- 본문 글자색: 진한 회색(#1f2937 수준)으로 충분한 대비 확보.
- 폰트: 시스템 기본 고딕(예: Pretendard, 맑은 고딕) — 가독성 우선.

> 실제 화면 시안 및 컴포넌트 디자인은 개발 단계에서 프론트엔드 디자인 작업으로 별도 제작한다. 본 기획서는 그 기준(토큰)을 정의한다.

## 11. 배포·운영 계획

### 11.1 GitHub
- 저장소 1개로 관리. `main`(운영), `dev`(개발) 브랜치 운영 권장.
- 비밀값(DB 접속, 관리자 비밀번호)은 코드에 넣지 않고 환경변수로 관리(`.env`는 커밋 제외).

### 11.2 Railway
- GitHub 저장소를 Railway에 연결 → `main` 푸시 시 자동 배포.
- PostgreSQL은 Railway 플러그인(관리형 DB)으로 추가, 접속 정보를 환경변수로 주입.
- 도메인은 Railway 기본 도메인 사용, 필요 시 커스텀 도메인 연결.

### 11.3 환경 변수(예)

```
DATABASE_URL=postgresql://...
ADMIN_USERNAME=...
ADMIN_PASSWORD=...
```

### 11.4 운영 절차
1. 설문 데이터(문항) 입력 → 2. 응답기간 설정(open) → 3. 문자로 URL 발송 → 4. 응답 수집 → 5. 종료(closed) → 6. 결과 집계·내보내기.

## 13. 향후 확장 계획

1. **설문 빌더**: 관리자가 화면에서 직접 새 설문/문항을 만드는 기능.
2. **중복 응답 방지 옵션**: 필요 시 수강생별 토큰 링크 발급.
3. **문자 발송 연동**: SMS API로 URL 자동 발송.
4. **설문 템플릿**: 만족도/수요조사/행사 신청 등 유형별 템플릿.
5. **결과 리포트 자동 생성**: 집계 결과를 PDF/문서로 자동 출력.

본 시스템의 DB·코드 구조(설문→문항→응답)는 위 확장을 코드 대규모 변경 없이 수용하도록 설계되어 있다.
